系统需求分析
系统需求分析是软件开发生命周期中的核心环节，旨在通过对业务逻辑、用户行为及系统约束的深度解构，为后续的架构设计与功能实现提供科学依据。本章将从业务需求、用户需求、功能性需求及非功能性需求四个维度展开，重点分析生鲜防伪场景下的数据流转与双重风控状态机的逻辑需求。
业务需求
从系统业务角度对生鲜防伪溯源系统进行分析，并设计数据流图，以图形化方式展示系统逻辑功能及数据在系统内的加工过程。通过自顶向下的分解方法，归纳出本系统涉及系统管理员、消费者两个主要实体，并描绘信息在生鲜供应链各环节中的流动与处理情况。

[此处插入图3-2 系统1层数据流图]
根软件工程要求，现对 1 层数据流图的重点加工条目进行详细说明：
1. 扫码查验与状态解析加工条目
加工名称：扫码查验与状态解析
描述：解析加密 URL 参数，动态计算商品实时生命周期
激发条件：消费者发起移动端 H5 查询请求
输入：AES 加密 uniqueCode
输出：物理状态结果及溯源档案
加工逻辑：系统接收请求后，利用 Redis 进行 scan_count 自增操作；随后通过“懒加载”机制，对比当前时间与“生产日期+保质期”，若超期则动态切入 Expired 状态。此过程与风控预警逻辑保持解耦，仅计算物理流通属性。
2. 异常频次风控预警加工条目
加工名称：异常频次风控预警
描述：基于 Redis 计数器进行数学阈值判定
激发条件：scan_count 发生变更
输入：实时累计扫码次数
输出：防伪预警工单记录
加工逻辑：当 scan_count $\ge$ 5 次（系统设定阈值）时，系统异步触发风控引擎，生成 Pending 状态的预警记录，而不直接锁定物理状态，实现物理与风控的双重解耦。
3. 预警人工干预加工条目
加工名称：预警人工干预
描述：管理员对风控异常进行仲裁
激发条件：管理员进入预警处理中心
输入：预警工单编号+干预指令（freeze/ignore）
输出：更新后的二维码物理状态或处理状态
加工逻辑：若管理员选择“冻结拦截（freeze）”，则跨表强制回写 status 为 Recalled，无视保质期流转，宣告该码物理死亡；若选择“忽略（ignore）”，则仅标记工单已完结，维持原物理状态流转。
用户需求
根据系统业务需求，系统大致分为两种实体：消费者实体、系统管理员实体。每种实体角色在系统中都有相应的功能需求与交互边界。
消费者实体是防伪验证与溯源信息获取的终端用户。其主要功能需求包括：通过移动端 H5 界面实现一键扫码查验、浏览产品全生命周期的溯源图文档案、实时查看商品的防伪风控警告（如红/黄/绿码提示）以及核对物理流转状态。

功能性需求
根据生鲜防伪溯源系统的业务建模分析，系统功能需求主要划分为以下四大核心服务模块。
（1）二维码加密与批次管理服务
该模块主要负责生鲜生产源头的数字化建模。支持管理员对供应商资质、产品规格及保质期等静态档案进行精细化管控 。核心功能包括调用 AES(高级加密标准) 对称加密框架，结合时间戳及随机扰动因素，将原始数据封装为不可逆推且长度均等的加密 unique_code 。同时集成 ZXing 引擎，将加密查验地址渲染为物理二维码图像，从根源切断黑客穷举 URL 路径的可能。
全链路溯源数据采集服务
本服务面向供应链中的各级操作员，提供全生命周期的流转节点上报功能 。模块支持在加工、质检、仓储、冷链运输及出库等关键环节，通过图文并茂的方式录入实时数据 。系统利用批次外键将碎片化的流转信息串联成垂直时间轴，确保生鲜食品从田间到餐桌的每一个动作皆可追溯、可审计。
移动端 H5 消费者扫码查询服务
作为面向终端用户的唯一交互界面，该服务采用轻量化 H5 响应式技术栈，实现跨平台的无缝接入 。当消费者扫码发起 GET 请求时，后台 API 网关迅速解密入参，匹配 MySQL 基础属性并结合 Redis 频次指标进行实时合成 。前端根据复合拦截指令，按需输出正常溯源图文、黄色高频预警或红色召回阻断视图。
智能防伪风控预警服务
这是系统的核心亮点，体现了“物理状态”与“风控状态”的双重解耦逻辑 。服务通过拦截器对 Redis 内存中的防刷计数器执行 INCR 原子操作，不依赖物理状态的改变即可监测扫码频度 。一旦探测到扫描频次超越设定的安全基线，系统自动实例化预警记录并推向管理员大盘，在不误伤正常物理流通的前提下，精准拦截黑灰产的克隆攻击。
非功能性需求
对于生鲜防伪溯源系统，不仅需要保证正常业务流转，还需满足高并发查询下的性能表现、系统安全性以及移动端的极致兼容性
。
（1）并发量与性能需求
由于生鲜扫码场景具有明显的瞬时高频特性（如超市促销期），系统必须引入 Redis 缓存机制作为前端查询的屏障 。通过 Redis 拦截并累加缓存频次，可有效缓解数据库 IO 压力，确保溯源查验及防伪判定的响应时间维持在毫秒级，为消费者提供流畅的查验体验。
（2）可用性与容错需求
系统需具备企业级的稳定运行能力。针对可能出现的并发峰值抖动，移动 H5 页面应设计降级方案，在服务超时或瞬时故障时呈现友好的安抚引导页 。核心数据库操作必须遵循 ACID 事务准则，确保物理生命周期回写及召回阻断指令的执行具备强一致性。
（3）安全性需求
在安全性是溯源系统的生命线。系统底层必须采用 AES 加密技术对 unique_code 进行处理，防范攻击者通过暴力破解进行参数穷举 。管理后台需实施严密的 RBAC（基于角色的访问控制），并在关键操作（如批次召回、防伪冻结）前通过二次密码或令牌校验，保障核心业务指令不被误触或越权执行。
（4）兼容性需求
考虑到消费者移动终端的多样性，表现层 H5 页面必须适配 iOS 与 Android 系统的主流内置扫码器。系统前端需经过多浏览器引擎（如 Webkit）的兼容性调优，确保垂直时间轴、预警警告卡片及质检图文在不同分辨率屏幕下均能精准渲染。
