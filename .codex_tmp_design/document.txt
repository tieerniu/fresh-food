系统设计
本章旨在根据前期系统需求分析的结果，确立“基于二维码的生鲜食品防伪溯源系统”的底层逻辑模型与工程实现基准。本章将详细论述系统在单体架构下的总体物理拓扑设计、底层安全与防伪密码学机制，并重点遵循数据库第三范式构建核心概念模型与物理数据表结构，为后续的系统编码与功能实现提供高度准确的架构指导。
系统总体架构设计
本系统遵循“高内聚、低耦合、轻量化”的软件工程设计原则，摒弃了状态开销繁重的微服务架构，全面采用基于 Spring Boot 的 B/S（浏览器/服务器）前后端分离单体架构。该架构不仅能够保障生鲜系统高频交互的绝对性能，同时极大降低了企业后期的运维部署成本。系统总体架构自上而下划分为三大核心层级：表现层、业务逻辑层与数据持久层。

系统安全机制设计
在生鲜防伪溯源系统的核心资产是数据的不可伪造性与权限的严密性。本系统从接口访问鉴权与底层密码学防伪两个维度，构建了轻量且坚固的安全防护体系。

系统底层安全机制的逻辑实现具体如下：
系统身份鉴权（MD5 盐值哈希与 UUID Token 机制）： 在后台管理人员的认证链路上，系统拒绝明文传输与存储。当管理员提交登录凭证时，业务层调用底层加密工具对明文密码进行 MD5 单向散列及加盐（Salt）混淆，随后与数据库存储的密码密文进行等值匹配，彻底规避了数据库被“拖库”后的凭证泄露风险。认证通过后，系统采用无状态的轻量级鉴权模式，通过 UUID.randomUUID() 剥离横线生成 32 位全局唯一 Token，作为会话的锚点下发至前端。此机制降低了单体架构下的 Session 内存开销，同时配合 Vue 3 的前端路由卫士实现了精准的页面级权限拦截。
防伪溯源码加密（动态 IV 向量与扰动熵混淆）： 这是切断黑产团伙通过穷举规律批量克隆二维码的“密码学物理长城”。系统在执行批量赋码逻辑时，摒弃了简单的自增 ID，而是通过定制化的算法原语生成密文。底层加密引擎提取当前系统的毫秒级时钟特征（System.currentTimeMillis()）作为动态初始化向量（IV），同时注入高强度随机扰动熵（Math.random()），混合输入算法引擎，最终输出长度恒定、完全无序且防推测的 unique_code。此外，系统在循环建码时额外颁发了 16 位截取长度的 UUID 作为防伪次级验证因子（security_token）。最终，这两项密文标识与外网校验路由拼接，经由 ZXing 矩阵压入流，渲染为不包含任何明文特征的 Base64 溯源图像，筑牢了防伪第一道防线。
数据库设计
数据库是生鲜防伪溯源双重风控状态机得以运转的基石。本节根据前文的数据流向与业务加工逻辑，严格按照第三范式要求进行数据库概念模型与物理表结构的设计。
概念模型设计
本
系统的核心概念模型高度贴合“源头建档 - 批量赋码 - 节点流转 - 扫码查验 - 风控报警”的生鲜供应链闭环。实体间的逻辑关系如下：
供应商（Suppliers）与产品批次（Product Batches）呈现 1:N（一对多）关系；产品批次与二维码（QR Codes）呈现 1:N 关系，同一批次可派生出成千上万个具备独立生命周期的单品密码；同时，产品批次与溯源流转记录（Traceability Records）及质检召回记录（Quality Inspections）呈 1:N 关系。在风控验证侧，单品二维码与扫码日志（Scan Logs）以及防伪预警记录（Security Warnings）分别呈现 1:N关系，以支撑双重解耦算法的逻辑判定。

系统用户与权限管理模块设计
功能描述
本模块主要负责企业后台管理员的数字身份生命周期管理与资源权限访问控制，保障溯源及预警管控后台不被非法侵入，维护企业核心数据的机密性与完整性。
算法描述
系统日志管理模块为例，详细论述其服务端具体运行逻辑。管理员在前端界面发起登录请求后，系统的 LoginController 接管会话。服务端首先提取传入的密码明文，调用内部工具类 MD5Utils.encrypt() 对其执行结合盐值（Salt）的单向混淆哈希运算，随后将其与 MySQL 中 users 表对应的 password_hash 字段进行等值安全比对。校验通过后，为降低会话状态开销，系统调用 UUID.randomUUID().toString() 去划线生成 32 位无状态 Token 令牌下发至客户端。在后续的 API 交互中，系统将基于 Token 映射验证操作员合法性，同时协同 Vue 3 前端的 Router Guards（路由卫士）共同构筑视图层的精细化权限拦截屏障。
供应商与生鲜批次模块设计
功能描述
本模块作为溯源数据流转的源头水库，提供合作基地建档以及对每日产线出仓生鲜单品进行批次化元数据归档管理的功能，确保“一物一码”具备明确的实体映射锚点。
算法描述
以业务专员在管理界面完整录入果种、原产地、实际生产日期（productionDate）及预设保质期天数（shelfLifeDays）等产品档案。该请求报文经前端封装后抛送至后端的对应 Controller。在参数合法性校验通过后，系统服务层利用 MyBatis-Plus 框架所提供的 insert() ActiveRecord 方法，将业务对象转化为底层 SQL 语句，持久化至 product_batches 数据表中。此操作为后续海量防伪二维码的生成提供了不可或缺的 batch_id 外键溯源基础。
溯源节点与质检召回模块设计
功能描述

防伪二维码加密与生成模块设计
功能描述
该模块为系统注入防伪基因。它承担着将静态的批次属性转化为成千上万个具备独立生命周期且绝对不可逆推、篡改的防伪溯源物理图像的重任。
算法描述
公后台管理员下达制码指令并选定规模后，前端 POST 请求至 /api/qrcode/generate。QrCodeController 解析出 batchId 并触发 for 循环架构。在每一次循环内，底层算法提取 System.currentTimeMillis() 毫秒级时钟因子，并混合 Math.random() 高强度随机扰动输入加密引擎，生成无序的 unique_code 标识；同时辅以 UUID 截断生成的 16 位 securityToken 作为防伪辅签。最终，系统整合密文 URL 并调用 ZXing 矩阵算法渲染出 Base64 格式的二维码图形包，并将这批具有独立 ID 的实体以 status = Active 态批量落库至 qr_codes 表中。

H5扫码端与双重风控模块设计（系统核心引擎）
（1）功能描述
这是连接消费者与防伪大脑的核心枢纽。模块不仅承担向消费者展示无感、流畅的生鲜流转时间轴的任务，更负责在高并发扫码行为中抽丝剥茧，拦截黑灰产团伙的“一码多印”及批量高频克隆探测攻击。

算法描述
订当终端发起的 GET 请求命中 TraceabilityController 后，系统引擎首先采用“极简懒加载防过期判断法”：通过 LocalDate.now().isAfter() 计算出库日期加上保质期是否超越当前系统时钟，若已超期且未处于被售或召回状态，则执行 updateById 迅速将物理状态截断降级至 Expired。 紧接着，控制层无条件对 Redis 内存池或数据库内的防重计数器发起 INCR 累加指令（scanCount + 1）。防伪衍生研判机制随即并发检测，若发现 scanCount >= 5（已破报警阈值），在执行幂等探雷操作确保无重复工单后，异步实例化 SecurityWarning 异常风控工单并追加持久化。最后，系统巧妙地将物理对象的生死属性（qrCodeStatus）与高频风控解耦组装成复合 JSON VO 报文，下推至 H5 前端引擎进行红、黄预警图层的动态叠加渲染。

防伪预警人工干预模块设计
（1）功能描述
作为自动风控防线的最后一道安全阀，该模块允许具有经验的安全管理员面对异常堆积的高频扫描工单进行研判裁决，确保算法误伤最小化的同时对确凿的克隆伪劣品执行物理核级抹杀。
（2）算法描述
产当安全专员向服务端 SecurityWarningController 抛送处理动作时，控制器深度解析该前端指令（Action）。若操作分支判定为冻结拦截（freeze），系统将以传入的工单 unique_code 为锁定信标，调用 UpdateWrapper 发动跨表级联打击，直接对底座 qr_codes 表中的目标记录实施强制覆盖，将 status 字段硬回写为最极端的死刑隔离态——Recalled。该动作实现深空物理阻断，此后全国范围扫该码均将面临红色封杀墙；若操作分支为误报忽略（ignore），则程序仅在本地工单域内将处理位变更为 Resolved 态归档，二维码在全网的自然流转进程将不受到任何代码层面的横向干扰。

本章小结
本章自顶向下地确立了基于 Spring Boot 单体架构的生鲜食品防伪溯源系统的总体物理设计基调，明确了 MD5 混淆与 AES 组合加密的系统底层安全屏障。通过高度规范的实体演化，系统完成了满足第三范式的 8 张核心 MySQL 物理表结构构建；在此数据底座之上，本章深入推进至代码方法级边界，详细解构了 6 大核心模块的功能与微观算法时序，尤其是将创新的“懒加载差集验证与 Redis 并发双重状态机风控解耦模型”落到了真实的编程实现链路上，为后续系统的编码落地与压力测试提供了极具严谨性的全套架构图纸。
