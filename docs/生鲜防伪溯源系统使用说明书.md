# 生鲜防伪溯源系统使用说明书

本文档用于说明《生鲜防伪溯源系统》的业务目标、角色分工、功能模块和完整使用流程。适用于系统开发者自测、毕业设计答辩演示、论文导师了解系统逻辑和普通管理员日常使用。

## 1. 系统简介

本系统是一个基于 Spring Boot + Vue 3 的生鲜产品防伪溯源管理平台，围绕“产品批次、溯源节点、二维码、防伪预警、质检召回”构建完整业务闭环。

系统的核心目标是：

- 管理生鲜产品批次和合作基地信息。
- 为产品批次生成防伪二维码。
- 消费者扫码后可在 H5 页面查看产品真伪和溯源信息。
- 系统记录扫码行为，当二维码被异常高频扫码时自动生成防伪预警。
- 管理员可对预警进行误报忽略、发起质检或冻结召回处理。
- 质检结果可反向联动二维码状态，形成“扫码发现风险 -> 预警 -> 质检 -> 召回/关闭”的闭环。

## 2. 系统角色说明

### 2.1 超级管理员 admin

超级管理员拥有系统最高权限，主要负责全局管理和风险处置。

可使用功能：

- 控制台数据总览
- 产品批次管理
- 合作基地管理
- 溯源记录管理
- 二维码管理
- 质检与召回
- 防伪预警中心

典型职责：

- 创建合作基地账号。
- 管理所有产品批次。
- 生成和维护二维码。
- 查看系统风险预警。
- 对风险二维码进行质检、冻结、召回。

### 2.2 合作基地 enterprise

合作基地是产品生产或流通环节的数据维护方。

可使用功能通常包括：

- 查看和维护自己名下的产品批次。
- 录入自己产品批次的溯源节点。
- 查看自己相关的数据。

注意：合作基地不能访问管理员专属风险处置、全局大盘和其他基地数据。

### 2.3 消费者 H5 扫码用户

消费者无需登录，只需要使用手机扫描二维码即可访问 H5 溯源页面。

消费者可以看到：

- 二维码认证状态
- 产品基本信息
- 生产批次信息
- 溯源时光轴
- 二维码是否召回或过期
- 系统记录的查询次数

消费者不能修改任何后台数据。

## 3. 推荐完整业务流程

建议按照以下顺序使用和演示系统，这条流程最能体现毕业设计的完整性。

1. 管理员登录后台。
2. 在“合作基地”中创建或确认合作基地账号。
3. 在“产品管理”中新增产品批次，填写产品名称、基地、产地、生产日期、保质期和批次数量。
4. 在“溯源记录管理”中选择该产品批次，依次录入生产、采摘、质检、包装、运输、销售等节点。
5. 在“二维码管理”中为该批次生成二维码，生成数量不能超过产品批次数量。
6. 使用手机扫描二维码，进入 H5 溯源页面查看产品认证和溯源信息。
7. 多次扫描同一个二维码，模拟异常高频扫码。
8. 系统在“防伪预警中心”生成预警。
9. 管理员根据预警选择误报忽略、发起质检或冻结召回。
10. 如果进入质检流程，在“质检与召回”中录入质检结论。
11. 若质检不合格，系统可联动召回二维码。
12. 回到控制台查看系统数据、扫码趋势、二维码状态和风险情况。

这条链路可以概括为：

```text
合作基地/管理员录入产品
-> 录入溯源节点
-> 生成二维码
-> 消费者扫码
-> 系统记录扫码行为
-> 高频扫码触发预警
-> 管理员处理预警
-> 质检确认
-> 召回或关闭风险
```

## 4. 功能模块使用说明

## 4.1 控制台

控制台用于查看系统整体运行状态，是管理员进入后台后的首页。

主要内容：

- 产品批次数量
- 溯源记录数量
- 二维码数量
- 今日查询次数
- 待处理预警
- 召回二维码数量
- 近 7 日扫码趋势
- 二维码状态分布
- 系统摘要

使用建议：

- 每次演示系统时先打开控制台，让导师看到系统具备数据总览能力。
- 如果待处理预警不为 0，应进入“防伪预警中心”处理。
- 如果召回二维码数量不为 0，应说明该数据来源于风险处置或质检召回。

## 4.2 产品管理

产品管理以“产品批次”为核心。系统不是只管理某一种产品，而是管理每一次生产或入库形成的批次。

新增批次时需要填写：

- 产品名称
- 所属基地
- 产地
- 生产日期
- 保质期
- 批次数量
- 图片链接
- 描述

重要说明：

- 批次号由系统自动生成，建议不要手动修改。
- 批次数量代表本批次最多可生成多少个二维码。
- 发码进度显示“已生成二维码数量 / 批次数量”。
- 如果发码进度已满，则不能继续生成二维码。

操作建议：

- 演示时可以创建一个小批次，例如批次数量设置为 5 或 10，方便生成和测试二维码。
- 不建议随意删除已有批次，尤其是已经生成二维码或已有溯源记录的批次。

## 4.3 合作基地

合作基地用于管理企业或生产基地账号。

主要字段：

- 基地名称
- 负责人
- 联系电话
- 地址
- 资质评级
- 企业账号
- 账号状态
- 关联批次

常用操作：

- 新增基地
- 编辑基地信息
- 重置密码
- 禁用或启用账号
- 删除基地

业务说明：

- 合作基地账号用于区分不同企业的数据归属。
- 一个基地可以关联多个产品批次。
- 禁用账号后，该企业不应继续进行后台操作。

## 4.4 溯源记录管理

溯源记录管理用于为产品批次录入完整的生产和流通过程。

页面采用“左侧批次目录 + 右侧时间线工作台”的方式：

- 左侧选择产品批次。
- 右侧查看该批次的溯源节点。
- 点击“录入下一节点”添加新节点。

标准节点顺序：

1. 种植
2. 采摘
3. 质检
4. 包装
5. 运输
6. 销售

每个节点建议填写：

- 溯源环节
- 操作员
- 地点
- 温度
- 详细描述

业务规则：

- 系统会按照标准顺序录入节点。
- 新增节点时，默认只能录入下一个应录入环节。
- 编辑节点时主要修改详情，不建议随意改变节点顺序。
- H5 消费者页面会展示这些节点，因此填写内容应简洁、真实、易懂。

示例节点内容：

```text
环节：种植
地点：咸阳基地
操作员：杨景瑞
温度：20℃
描述：完成苹果种植登记，生长状态正常。
```

## 4.5 二维码管理

二维码管理用于查看和维护每个产品批次的二维码。

当前页面采用“批次总览 + 二维码明细抽屉”设计。

主页面展示：

- 二维码总数
- 激活码数量
- 召回码数量
- 过期码数量
- 风险码数量
- 每个批次的发码进度
- 每个批次的状态分布
- 每个批次的扫码表现
- 每个批次是否存在风险码

点击“查看二维码”后，可以进入该批次的二维码明细。

二维码明细中可操作：

- 查看二维码图片
- 下载二维码图片
- 标记为召回
- 标记为过期
- 删除二维码

二维码状态说明：

| 状态 | 含义 | H5 页面表现 | 后台处理建议 |
| --- | --- | --- | --- |
| 激活 | 二维码正常可用 | 显示正品认证和溯源信息 | 正常使用 |
| 召回 | 二维码对应产品存在风险 | 提示产品已召回或存在风险 | 通常由预警或质检触发 |
| 过期 | 二维码对应产品超过有效期 | 提示产品过期 | 可由保质期或管理员操作触发 |

注意：系统中不再使用“已售”状态。二维码状态严格保持为“激活、召回、过期”三类。

生成二维码规则：

- 必须先有产品批次。
- 生成数量不能超过批次剩余可发码数量。
- 每个二维码都有唯一防伪码。
- 生成后可下载图片，用于手机扫码测试。

## 4.6 质检与召回

质检与召回模块用于记录产品批次的质量检测结果，并根据结果决定是否召回。

质检来源：

- 管理员手动新增质检。
- 防伪预警中心发起质检后自动关联生成。

质检结果：

- 合格：说明风险已核验通过，可关闭相关风险。
- 不合格：说明产品存在质量风险，应启动召回。

业务建议：

- 如果某个二维码高频扫码，但无法判断是否真实风险，可以先发起质检。
- 如果质检不合格，应将相关二维码或批次升级为召回状态。
- 召回后，消费者再次扫码会看到异常提示。

## 4.7 防伪预警中心

防伪预警中心用于处理异常扫码产生的风险工单。

预警来源：

- 同一个二维码被频繁扫描。
- 系统判断存在恶意复制或异常访问风险。

预警状态：

- 待处理：系统刚生成，管理员尚未处理。
- 待质检：管理员已发起质检，等待质检结论。
- 已处理：风险已关闭、忽略或召回。

处理方式：

- 误报忽略：认为是测试或正常扫码，不进入质检和召回。
- 发起质检：创建质检任务，由质检结果决定后续处理。
- 冻结拦截：立即将二维码升级为召回状态，适用于明显异常风险。

使用建议：

- 演示时可以用同一个二维码连续扫码，观察是否生成高频预警。
- 如果是测试数据，可以选择“误报忽略”。
- 如果想展示完整闭环，应选择“发起质检”，再到质检模块录入结果。

## 4.8 H5 消费者扫码页面

H5 页面是消费者扫码后看到的页面，是系统面向普通用户的入口。

消费者扫码后会看到：

- 正品认证或异常提示
- 产品名称
- 生产批次
- 产地
- 生产日期
- 保质期
- 溯源时光轴
- 扫码查询次数

H5 页面状态说明：

- 激活码：显示正品认证和溯源信息。
- 召回码：提示该产品已召回或存在风险。
- 过期码：提示该产品已过期。
- 无效码：提示二维码不存在或非法。

使用手机测试时需要注意：

- 手机和电脑应连接同一个 WiFi。
- 二维码中的 H5 地址必须是电脑在当前局域网中的 IP 地址。
- 如果电脑切换 WiFi 或 IP 变化，需要重新确认移动端入口地址。

## 5. 推荐演示流程

下面是一套适合给导师演示的流程。

### 5.1 正常溯源演示

1. 登录管理员后台。
2. 打开产品管理，新增一个产品批次。
3. 打开溯源记录管理，为该批次录入 2 到 3 个节点。
4. 打开二维码管理，为该批次生成 1 个二维码。
5. 点击查看二维码或下载二维码。
6. 手机扫码进入 H5 页面。
7. 向导师说明消费者可以看到产品信息和溯源时光轴。

### 5.2 防伪预警演示

1. 使用同一个二维码连续扫码多次。
2. 回到后台防伪预警中心。
3. 查看是否生成高频扫码预警。
4. 点击处理预警。
5. 选择发起质检或冻结拦截。
6. 如果选择冻结拦截，回到二维码管理查看该二维码状态变为召回。
7. 再次扫码，观察 H5 页面异常提示。

### 5.3 质检召回闭环演示

1. 在防伪预警中心选择“发起质检”。
2. 进入质检与召回模块。
3. 编辑质检任务并录入质检结果。
4. 如果结果为不合格，启动召回。
5. 回到二维码管理和控制台，查看召回状态变化。
6. 再次扫码，验证消费者端展示召回提示。

## 6. 常见问题说明

### 6.1 为什么手机扫码打不开页面？

可能原因：

- 手机和电脑不在同一个 WiFi 下。
- 二维码中的 IP 地址不是当前电脑 IP。
- 前端服务没有启动。
- 防火墙拦截了访问。

处理方法：

- 确认电脑和手机在同一局域网。
- 查看二维码管理页面顶部的“当前移动端入口”。
- 如果 IP 变化，重新生成或重新下载二维码测试。

### 6.2 为什么二维码不能继续生成？

原因通常是该批次二维码已经达到批次数量上限。

例如：

```text
批次数量：20
已生成二维码：20
剩余可发码：0
```

此时系统会禁止继续生成二维码，这是为了防止一个批次生成超过实际数量的防伪码。

### 6.3 为什么二维码只有激活、召回、过期？

这是系统当前设计的正式状态机。

- 激活：正常流通。
- 召回：存在风险或已被管理员冻结。
- 过期：超过有效期或管理员标记过期。

系统不使用“已售”状态，避免和生鲜产品溯源、防伪预警逻辑冲突。

### 6.4 高频扫码一定代表是假货吗？

不一定。高频扫码只能说明存在风险，例如：

- 消费者反复刷新。
- 管理员测试扫码。
- 二维码被截图传播。
- 二维码被复制盗用。

因此系统不会自动判定假货，而是先生成预警，再由管理员选择忽略、质检或冻结。

### 6.5 质检是否必须上传报告？

当前系统以录入质检结论为主，不强制上传报告文件。

毕业设计中可以说明：

- 当前版本支持质检结果、质检员、日期和备注记录。
- 后续可扩展上传 PDF、图片或检测机构报告编号。

这样既符合本科毕设范围，也保留了扩展方向。

## 7. 数据维护建议

为了让系统演示更清晰，建议保留少量高质量测试数据。

推荐数据结构：

- 2 个合作基地。
- 3 到 5 个产品批次。
- 每个批次 2 到 6 个二维码。
- 每个批次至少 2 个溯源节点。
- 至少 1 条待处理预警。
- 至少 1 条质检记录。
- 至少 1 个召回二维码。

不建议：

- 生成过多无意义二维码。
- 批次名称随意填写。
- 溯源节点描述写“哈哈哈”等测试文本。
- 删除已经联动预警或质检的数据。

## 8. 论文中可描述的系统亮点

本系统可以在论文中突出以下亮点：

- 使用 RBAC 区分管理员、合作基地和消费者权限。
- 使用产品批次作为溯源和二维码发放的核心业务对象。
- 使用二维码唯一防伪码实现消费者无登录查询。
- 使用扫码日志记录消费者查询行为。
- 使用高频扫码规则生成防伪预警。
- 使用预警、质检、召回形成闭环式风险处置。
- 使用二维码状态机区分激活、召回和过期。
- 后台管理端与 H5 消费者端数据联动。
- 前端页面按照产品级后台进行了信息聚合和操作分层。
